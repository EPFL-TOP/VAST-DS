Overview

VAST-DS is a high-throughput zebrafish drug-screening tool. It combines a Django web app for well-plate mapping and image browsing, Bokeh dashboards for interactive review and annotation, and three PyTorch models that score VAST-microscope images for fish validity, somite counts, and orientation.

The two Django apps live at well_mapping/ (experiments, plates, drugs, wells) and well_explorer/ (image dashboards, statistics, drug plots). Deep-learning training and inference live in somiteCounting/, with all preprocessing centralised in somiteCounting/_common.preprocess_image().

Deep-learning models

Three independent ResNet18-based models, all sharing the same preprocessing (1st/99th percentile clip + resize to 224×224, defined once in somiteCounting/_common.preprocess_image).

Model classInputOutputLoss
SomiteCounter_freeze YFP image (n_total_somites, n_bad_somites) float Weighted MSE
FishQualityClassifier YFP image valid logit BCE
OrientationClassifier BF image correct_orientation logit BCE

At inference, somite predictions are cast to non-negative integers via clamp_somite_count() before being saved to DestWellPropertiesPredicted.

Training data folder layout

Each training script expects:

<input_data_path>/
├── train/
│   ├── img001_YFP.png
│   ├── img001_YFP.json
│   └── ...
├── valid/   ← used for early stopping
└── test/    ← held-out, never seen during training; used only for the final report

Required JSON keys depend on the task — see README.md for the exact schema. A few key points:

  • Somite counter and validity classifier read images whose filename contains YFP.
  • Orientation classifier reads images whose filename does not contain YFP (i.e. brightfield).
  • To populate test/: use the resplit_training_data management command (below) — it assigns each annotated well to one of train / valid / test, sets the corresponding flag on DestWellProperties (use_for_training / use_for_validation / use_for_test), and rebuilds the on-disk folders to match. The wells with use_for_test=True show up in the Test (honest) column on the Statistics page.

One-shot re-split with resplit_training_data

A Django management command resets every annotation's three subset flags, randomly assigns each eligible annotation to train / valid / test (default 70 / 15 / 15), updates the database, and rebuilds the on-disk train/ / valid/ / test/ folders by copying source images and writing JSON sidecars.

Dry-run first (no DB writes, no file moves) to see what it will do:

python manage.py resplit_training_data --dry_run

Real run with a fixed seed for reproducibility:

python manage.py resplit_training_data \
    --output_path D:\vast\VAST-DS\training_data \
    --train 0.70 --valid 0.15 --test 0.15 \
    --seed 42 \
    --use_corrected

--use_corrected tells it to pull source images from each well's corrected_orientation/ subfolder (recommended after retraining the orientation classifier — see step 2 below). By default, annotations with valid=False are skipped, matching how the training scripts filter their datasets; pass --keep_invalid to include them.

Retraining

Order matters. Retrain the orientation classifier first, then re-run orientation correction on every well, then retrain the somite counter and validity classifier on the freshly canonicalised images. The somite/validity training data is typically sourced from each well's corrected_orientation/ subfolder, so refreshing those folders with the new orientation model produces cleaner training data.
  1. Retrain orientation: 
    python -m somiteCounting.training_orientation \
    --input_data_path D:\vast\VAST-DS\training_data \
    --model_save_path checkpoints \
    --epochs 40 --batch_size 8
  2. Re-run orientation correction on all experiments so every well's corrected_orientation/ folder uses the new model: 
    python somiteCounting/orientfish.py
    This walks every well, picks the best of {no flip, hflip, vflip, hflip+vflip}, and writes the flipped images to <well>/corrected_orientation/.
  3. Re-split annotations into train / valid / test and rebuild the on-disk training folders from the freshly canonicalised images (this also sets the use_for_training / use_for_validation / use_for_test flags on DestWellProperties): 
    python manage.py resplit_training_data \
    --output_path D:\vast\VAST-DS\training_data \
    --train 0.70 --valid 0.15 --test 0.15 \
    --seed 42 \
    --use_corrected
  4. Retrain the somite counter: 
    python -m somiteCounting.training \
    --input_data_path D:\vast\VAST-DS\training_data \
    --model_save_path checkpoints \
    --epochs 150 --batch_size 8 --patience 7
  5. Retrain the validity classifier: 
    python -m somiteCounting.training_valid \
    --input_data_path D:\vast\VAST-DS\training_data \
    --model_save_path checkpoints \
    --epochs 50 --batch_size 8
  6. Restart the Django server so the dashboard picks up the new checkpoints. Checkpoint paths are hard-coded in well_explorer/views.py (lines ~50–75) — if you change --model_save_path, update them too.

Each training run drops a *_test_metrics.json next to the checkpoint with the held-out report (MAE / RMSE / bias / R² for the somite counter; accuracy / precision / recall / F1 / confusion-matrix counts for the classifiers). That file is what to quote when comparing runs.

Why retrain right now? Until recently, each script had its own image normalisation: somite + validity used img /= img.max() (one hot pixel ruins contrast), orientation used percentile-clipping (robust), and dashboard inference used yet a third variant. After the _common.preprocess_image consolidation, all three models share the percentile-clipping path during both training and inference. Existing checkpoints will still load (the architecture didn't change) but their accuracy may drift because they were trained on a different colour space. Retraining once on the unified pipeline establishes a clean baseline going forward.

Roadmap

Planned, in order:

  1. Multi-row predictions schema — change DestWellPropertiesPredicted from a OneToOne to a multi-row table keyed by (dest_well, model_name, model_version) so multiple models (current ResNet, future SAM-based segmenter, etc.) can store predictions for the same well side by side. Stats and drug-plot pages gain a "which model?" selector.
  2. Per-somite data — initially as a JSONField on the prediction row (centroid, area, anterior-posterior position, defective flag per somite), promoted to a dedicated SomiteInstance model when cross-well queries require it.
  3. SAM segmentation dashboard — new Bokeh page where the user clicks somites to seed point prompts; MedSAM produces per-somite masks; an "auto-segment" button runs grid-prompt + NMS for batch processing. Predictions written to the multi-row schema with model_name='sam_v1'.
  4. Cross-model comparison UI — the prediction-vs-annotation table and the drug plot both gain dropdowns to select which model_name to compare against the manual annotations. Scatter plots can overlay multiple models for the same wells.
