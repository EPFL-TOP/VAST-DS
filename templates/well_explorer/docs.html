Overview

VAST-DS is a high-throughput zebrafish drug-screening tool. It combines a Django web app for well-plate mapping and image browsing, Bokeh dashboards for interactive review and annotation, and three PyTorch models that score VAST-microscope images for fish validity, somite counts, and orientation.

The two Django apps live at well_mapping/ (experiments, plates, drugs, wells) and well_explorer/ (image dashboards, statistics, drug plots). Deep-learning training and inference live in somiteCounting/, with all preprocessing centralised in somiteCounting/_common.preprocess_image().

Deep-learning models

Three independent ResNet18-based models, all sharing the same preprocessing (1st/99th percentile clip + resize to 224×224, defined once in somiteCounting/_common.preprocess_image).

Model classInputOutputLoss
SomiteCounter_freeze YFP image (n_total_somites, n_bad_somites) float Weighted MSE
FishQualityClassifier YFP image valid logit BCE
OrientationClassifier BF image correct_orientation logit BCE

At inference, somite predictions are cast to non-negative integers via clamp_somite_count() before being saved to DestWellPropertiesPredicted.

Training data folder layout

Each training script expects:

<input_data_path>/
├── train/
│   ├── img001_YFP.png
│   ├── img001_YFP.json
│   └── ...
├── valid/   ← used for early stopping
└── test/    ← held-out, never seen during training; used only for the final report

Required JSON keys depend on the task — see README.md for the exact schema. A few key points:

  • Somite counter and validity classifier read images whose filename contains YFP.
  • Orientation classifier reads images whose filename does not contain YFP (i.e. brightfield).
  • To populate test/: use the resplit_training_data management command (below) — it assigns each annotated well to one of train / valid / test, sets the corresponding flag on DestWellProperties (use_for_training / use_for_validation / use_for_test), and rebuilds the on-disk folders to match. The wells with use_for_test=True show up in the Test (honest) column on the Statistics page.

One-shot re-split with resplit_training_data

A Django management command resets every annotation's three subset flags, randomly assigns each eligible annotation to train / valid / test (default 70 / 15 / 15), updates the database, and rebuilds the on-disk train/ / valid/ / test/ folders by copying source images and writing JSON sidecars.

Dry-run first (no DB writes, no file moves) to see what it will do:

python manage.py resplit_training_data --dry_run

Real run with a fixed seed for reproducibility:

python manage.py resplit_training_data \
    --output_path D:\vast\VAST-DS\training_data \
    --train 0.70 --valid 0.15 --test 0.15 \
    --seed 42 \
    --use_corrected

--use_corrected tells it to pull source images from each well's corrected_orientation/ subfolder (recommended after retraining the orientation classifier — see step 2 below). By default, annotations with valid=False are skipped, matching how the training scripts filter their datasets; pass --keep_invalid to include them.

Retraining

Order matters. Retrain the orientation classifier first, then re-run orientation correction on every well, then retrain the somite counter and validity classifier on the freshly canonicalised images. The somite/validity training data is typically sourced from each well's corrected_orientation/ subfolder, so refreshing those folders with the new orientation model produces cleaner training data.
Strict rule on invalid fish. Annotations with valid=False are never used in training of any task. resplit_training_data and the dashboard's "Create Training Set" button both enforce this. Consequence: the validity classifier sees only positive examples and will degenerate — flag this when planning to retrain it on its own.
  1. Retrain orientation: 
    python -m somiteCounting.training_orientation \
    --input_data_path D:\vast\VAST-DS\training_data \
    --model_save_path checkpoints \
    --epochs 40 --batch_size 8
  2. Refresh corrected_orientation/ for every well using the new model. Wraps orientfish.py as a management command with proper CLI flags and an experiment filter: 
    python manage.py refresh_orientation \
    --checkpoint checkpoints/orientation_best.pth
# or limit to a single experiment:
python manage.py refresh_orientation --experiment VAST_2026-04
  3. Re-split annotations into train / valid / test and rebuild the on-disk training folders from the freshly canonicalised images (this also sets the use_for_training / use_for_validation / use_for_test flags on DestWellProperties). Includes any annotation with at least one usable label (somite or orientation), so orientation-only annotations get fair coverage: 
    python manage.py resplit_training_data \
    --output_path D:\vast\VAST-DS\training_data \
    --train 0.70 --valid 0.15 --test 0.15 \
    --seed 42 \
    --use_corrected
  4. Retrain the somite counter: 
    python -m somiteCounting.training \
    --input_data_path D:\vast\VAST-DS\training_data \
    --model_save_path checkpoints \
    --epochs 150 --batch_size 8 --patience 7
  5. Retrain the validity classifier: 
    python -m somiteCounting.training_valid \
    --input_data_path D:\vast\VAST-DS\training_data \
    --model_save_path checkpoints \
    --epochs 50 --batch_size 8
  6. Refresh production predictions for every dest well in the database with the new checkpoints (batch counterpart to the dashboard's "Predict Full Plate" button): 
    python manage.py reinfer
# or limit:
python manage.py reinfer --experiment VAST_2026-04 --plate 2
  7. Restart the Django server so the dashboard's in-memory model instances pick up the new checkpoints. Checkpoint paths are hard-coded in well_explorer/views.py (lines ~50–75) — if you changed --model_save_path, update them too.

Each training run drops a *_test_metrics.json next to the checkpoint with the held-out report (MAE / RMSE / bias / R² for the somite counter; accuracy / precision / recall / F1 / confusion-matrix counts for the classifiers). That file is what to quote when comparing runs.

Why retrain right now? Until recently, each script had its own image normalisation: somite + validity used img /= img.max() (one hot pixel ruins contrast), orientation used percentile-clipping (robust), and dashboard inference used yet a third variant. After the _common.preprocess_image consolidation, all three models share the percentile-clipping path during both training and inference. Existing checkpoints will still load (the architecture didn't change) but their accuracy may drift because they were trained on a different colour space. Retraining once on the unified pipeline establishes a clean baseline going forward.

Predictions schema

DestWellPropertiesPredicted is keyed by (dest_well, model_name, model_version) — multiple models can store predictions for the same well side-by-side. Read with latest_prediction(dest_well, model_name='resnet_v1') from well_mapping.models; write with DestWellPropertiesPredicted.objects.update_or_create(...) passing all three keys.

  • model_name: e.g. 'resnet_v1' (current ResNet18 stack), 'sam_v1' (incoming MedSAM segmenter), future 'multitask_v1'.
  • model_version: optional checkpoint tag (git hash, date). Empty string overwrites the previous row for that well/model on each run; a non-empty tag keeps history.
  • predicted_at: auto-updated; used by latest_prediction() to pick the most recent row.
  • per_somite_data: JSONField, used by SAM. List of dicts: 
    [{"index": 0, "centroid_x": 320, "centroid_y": 110,
  "area": 850, "ap_position": 0.42,
  "severity": 0, "comments": ""}, ...]
    Severity scale: 0 = healthy, 1 = mild, 2 = moderate, 3 = severe. AP position is normalised so 0 = head, 1 = tail.

Existing rows from before this schema change kept their data and got tagged model_name='resnet_v1' (Django populated this from the new field's default when the column was added). They have empty model_version='', same as new rows the dashboard writes. The schema migration is in well_mapping/migrations/0036_predictions_multirow.py.

SAM segmentation

SAM segmentation dashboard for per-somite analysis. Click each somite to seed a point prompt, press Segment, review the per-somite table, then Save. Stored under model_name='sam_v1' with per-somite data in per_somite_data.

Setup

SAM is loaded lazily on the first Segment click. Two requirements:

  1. Install Meta's segment_anything package: 
    pip install git+https://github.com/facebookresearch/segment-anything.git
  2. Drop a checkpoint file at checkpoints/sam_vit_b_01ec64.pth (or set the VAST_SAM_CHECKPOINT environment variable to a custom path). Smallest official weights are sam_vit_b (~350 MB) — download from the Meta SAM repo. vit_l and vit_h are larger but more accurate; set VAST_SAM_MODEL_TYPE accordingly.

MedSAM also works through this code path — drop the MedSAM checkpoint at the same path and keep vit_b as the model type.

Workflow

  1. Pick experiment, plate, well from the selectors. The canonicalised YFP image (<well>/corrected_orientation/) loads in the figure. If no canonicalised image exists yet, run python manage.py refresh_orientation first.
  2. Click on each somite in the image. Each click adds a red dot (positive point prompt).
  3. Press Segment. SAM runs once per prompt, producing one mask per somite. The masks overlay the image with one colour per somite, and the per-somite table fills with centroid / area / AP-position.
  4. Press Save. A row is upserted into DestWellPropertiesPredicted with model_name='sam_v1'; n_total_somites is the click count and per_somite_data contains the full per-somite list. Re-saving overwrites the row.
  5. Clear prompts resets clicks and overlays without touching the database.

Severity is initially 0 (healthy) for every somite. The editable severity UI and grid-prompt auto-segment are planned for a follow-up.

Roadmap

Planned, in order:

  1. [done] Multi-row predictions schema — see above.
  2. SAM segmentation dashboard — new Bokeh page where the user clicks somites to seed point prompts; MedSAM produces per-somite masks; an "auto-segment" button runs grid-prompt + NMS for batch processing. Predictions written with model_name='sam_v1' and per-somite results stored in per_somite_data.
  3. Cross-model comparison UI — the prediction-vs-annotation table and the drug plot both gain dropdowns to select which model_name to compare against the manual annotations. Scatter plots can overlay multiple models for the same wells.
  4. Per-somite as a dedicated model — promote per_somite_data JSON to a SomiteInstance table once cross-well queries on individual somites become routine.
